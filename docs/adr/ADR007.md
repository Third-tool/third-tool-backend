# ADR007: BC 간 협력에 동기 도메인 이벤트를 도입한다

- **상태**: Accepted
- **날짜**: 2026-05-14
- **관련**: [ADR005](ADR005.md), `docs/PACKAGE.md` §6 BC 간 의존 규칙

## 컨텍스트

Story-005-1 — 학습 자료(`LearningMaterial`) 등록 시 동명의 `Deck`이 자동 생성되어 축에 귀속되어야 한다 (Epic 5). LearningFacade BC가 Deck BC와 처음으로 협력하는 흐름이며, 다음 두 제약이 동시에 걸린다.

1. **동기 트랜잭션 보장 필요** — Story 본문이 "동명 Deck 존재 시 확인 팝업"을 요구한다. 자료 등록 응답에 `409 + DECK_NAME_DUPLICATE`를 즉시 반환해야 하며, 자료 저장도 같은 트랜잭션에서 롤백되어야 한다.
2. **BC 결합도 최소화** — 기존 BC 협력은 `Card → Deck`, `Review → Card·Deck`처럼 Application Service에서 Repository/QueryService를 직접 주입하는 패턴이었다. 그러나 Epic 5 메모는 "이벤트 기반 단방향 연동"을 명시했고, 자료 등록 흐름이 Deck 생성을 "요청"하는 의도(자료가 Deck 생성의 진입점)를 명확히 표현하고 싶다.

여기서 일반적 도메인 이벤트 패턴(`@TransactionalEventListener(AFTER_COMMIT)`)은 두 번째 요구만 만족시킨다 — 비동기 처리라 자료 응답에 동명 충돌 결과를 줄 수 없다.

또 부차적으로, 자료 등록 응답이 생성된 Deck의 `deckId`/`deckName`을 포함해야 한다 (FE가 "Deck으로 이동" 버튼에 사용). 비동기 이벤트로는 호출자가 결과를 받을 채널이 없다.

## 결정

**Spring `@EventListener` 동기 도메인 이벤트**로 BC 협력을 구현한다.

- 이벤트는 발행 BC(`LearningFacade/domain/event/LearningMaterialCreatedEvent`)에 정의한다. 수신 BC(Deck)가 import한다 — 단방향 의존 `Deck → LearningFacade`.
- 핸들러는 수신 BC의 application 계층에 둔다 (`Deck/application/event/LearningMaterialCreatedEventHandler`).
- `@TransactionalEventListener`가 **아닌** `@EventListener`를 사용해 호출자(LearningMaterialCommandService) 트랜잭션 안에서 동기 실행한다. 핸들러가 throw하면 자료 트랜잭션 전체가 롤백된다.
- 이벤트 객체는 **mutable 일반 class**로 만들고 핸들러가 `setResult(deckId, deckName)`로 결과를 설정한다. 호출자는 발행 직후 이벤트 객체에서 결과를 읽어 응답에 사용한다 — 동기 이벤트의 특성을 활용한 결과 통신 채널.

## 결과 (Consequences)

**긍정적**

- BC 간 협력 의도가 코드 레벨에서 명시적으로 분리된다 — 자료 등록 Service는 "이벤트 발행"만 알면 되고, Deck 생성 세부 로직(동명 검사·suffix 부여)은 Deck BC 안에 응집된다.
- 동기 처리로 동명 충돌이 자료 응답에 즉시 반영된다 (`@TransactionalEventListener`의 비동기 한계 회피).
- 같은 트랜잭션이라 원자성 보장 — 자료 저장 후 Deck 저장 실패 시 자료도 함께 롤백.
- 호출자가 이벤트 객체에서 결과를 읽어 응답을 빌드 — Deck BC가 Response DTO를 알 필요 없음 (양방향 의존 회피).

**트레이드오프 / 부정적**

- **이벤트 객체가 mutable** — 일반적 도메인 이벤트 표준(immutable record)에서 벗어남. 동기 + 결과 통신 패턴에서만 안전하며, 비동기 처리 시 race condition 위험. 본 패턴을 비동기로 전환할 경우 결과 통신 메커니즘을 재설계해야 한다.
- **새 BC 의존 도입** — Deck → LearningFacade (이벤트 import). `docs/PACKAGE.md` §6에 새 행 추가 필요.
- **핸들러가 throw하는 예외가 호출자에 그대로 전파** — 핸들러는 `BusinessException(DECK_NAME_DUPLICATE)`를 throw하고, 호출자(LearningFacade BC)의 GlobalExceptionHandler가 그것을 409로 매핑. 결과적으로 LearningFacade BC가 발행한 이벤트가 Deck BC 코드 경로의 예외로 끝나며, 디버깅 시 스택 트레이스를 BC 경계 두 번 가로지른다.
- **Spring 이벤트 인프라 도입** — 코드베이스 첫 사례. 다른 BC 협력 패턴(직접 주입)과 다른 모델이 공존.

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| 직접 주입 (`DeckCommandService` 또는 `DeckQueryService` 주입) | 기존 BC 협력 패턴과 일관. 결과 통신 자연스러움 | Epic 5 메모의 "이벤트 기반 단방향" 의도와 어긋남. BC 협력 의도가 코드에 덜 명시적 |
| `@TransactionalEventListener(AFTER_COMMIT)` 비동기 이벤트 | 일반 도메인 이벤트 표준 | 자료 응답에 409를 줄 수 없음 — Story 요구 미충족. 응답에 deckId 포함 불가 |
| Coordinator Service 신설 (`MaterialDeckCoordinator`) | 양쪽 BC의 협력 의도를 한 객체에 응집 | 새 추상화 도입. BC 둘 다 알아야 해 결합도 더 높음. 본 Story 한정으로 무게 과함 |
| 이벤트 record(immutable) + 호출자가 Repository로 결과 재조회 | 이벤트 표준 보존 | LearningFacade BC가 DeckRepository 직접 주입 → BC 경계 위반 또는 DeckQueryService 메서드 추가 (작업 증가) |

## 다시 검토할 시점

- **BC 협력이 3개 이상 본 패턴(`@EventListener` 동기 + mutable 이벤트)을 따르게 된 시점** — 패턴이 표준화되었음을 의미. 가이드라인 문서(`docs/PACKAGE.md` BC 협력 섹션)에 정착시키고, mutable 이벤트의 안전 운용 규칙을 정의한다.
- **비동기 처리 요구가 발생한 시점** (예: 자료 등록 후 외부 API 호출, 메일 발송 등) — 본 결정의 결과 통신 메커니즘이 깨지므로 별도 ADR로 비동기 패턴을 분리 결정한다.
- **BC 의존 그래프가 복잡해져 사이클 위험이 보이는 시점** — Deck → LearningFacade 의존이 다른 BC와 결합되어 양방향이 되지 않는지 점검한다.
- **이벤트 객체에 결과 외 부수 효과(예: 외부 시스템 ID, audit 정보)가 추가되기 시작한 시점** — 이벤트가 단순 통보가 아닌 "협력 결과 컨테이너"로 변해 SRP를 잃을 수 있다. Coordinator 패턴 재검토.
