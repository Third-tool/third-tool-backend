# ADR003: Soft Delete는 사용자 자산성 도메인에만 선택 적용한다

- **상태**: Accepted
- **날짜**: 2026-05-07
- **관련**: [`private-docs/table/learningFacade.md`](../../private-docs/table/learningFacade.md) §3-4, [`private-docs/domain/card.md`](../../private-docs/domain/card.md), [`private-docs/domain/deck.md`](../../private-docs/domain/deck.md)

## 컨텍스트

도메인마다 삭제 의미가 다르다. 사용자가 "이건 진짜 지우는 게 아니라 잠시 내려놓는 행위"라고 인식하는 도메인이 있고, "구조를 재구성하는 편집 행위"라고 인식하는 도메인이 있다.

Card·Deck·LearningFacade·LearningMaterial은 **사용자가 시간을 들여 만든 자산**이다. 잘못 삭제했을 때 복구를 원할 가능성이 높고, 활동 패턴 분석 가치도 있다.

LearningAxis·AxisTopic·TopicMaterial·CardTag·KeywordCue는 **자유롭게 재구성하는 구조 편집 단위**다. 책 목차를 다시 짜듯 추가·삭제가 자연스러운 편집 액션이고, 복원 요구가 사실상 없다.

운영 환경 MySQL 8.0은 PostgreSQL의 부분 인덱스(`WHERE deleted_at IS NULL`)를 지원하지 않아, soft delete를 광범위하게 적용하면 모든 인덱스의 카디널리티가 활성+삭제 데이터로 부풀어 오른다.

## 결정

**적용 대상** (`deleted_at DATETIME(6) NULL` + `@SQLRestriction` + `@SQLDelete`):

| 테이블 | 근거 |
| --- | --- |
| `card` | 사용자가 시간을 들여 만든 학습 자산. 복구 요구 가능 |
| `deck` | 카드 컨테이너. 잘못된 삭제 복원 가치 |
| `learning_facade` | 컨셉·축·자료의 루트. 전체 학습 맥락 |
| `learning_material` | 사용자가 직접 등록한 외부 자료 |
| `users` | 계정 자체 |

**미적용 대상** — 일반 DELETE 또는 `orphanRemoval`로 처리:

| 테이블 | 근거 |
| --- | --- |
| `learning_axis`, `axis_topic` | 책 목차처럼 동적으로 재구성되는 편집 단위 |
| `topic_material`, `card_tag` | 연결 사실 기록 — 해제는 사실의 종료 |
| `keyword_cue` | Card에 종속된 자식, Card 삭제 시 함께 처리 |
| `card_status_history` | 이력 자체. 삭제는 Card 삭제 시 cascade |

**JPA 매핑 표준**:

```java
@Entity
@SQLRestriction("deleted_at IS NULL")
@SQLDelete(sql = "UPDATE table_name SET deleted_at = NOW(6) WHERE id = ?")
public class XxxEntity { ... }
```

하드 삭제는 금지. 도메인은 `softDelete()` 메서드를 통해서만 제거 의도를 표현한다.

**물리 삭제 정책**: `deleted_at`이 설정된 지 90일 경과한 행은 월 1회 배치로 `archive` 스키마로 이동 후 본 테이블에서 DELETE한다. 현재 데이터 규모가 작아 즉시 도입은 불필요 — 6개월 이상 운영 후 규모 확인 뒤 결정.

## 결과

**긍정적**
- 사용자 복원 요청에 즉시 대응 가능.
- 활동 패턴 분석·감사 추적에 자료 보존.
- 구조 편집 도메인은 일반 DELETE의 단순성을 유지 — 인덱스·쿼리 부담 최소화.

**트레이드오프**
- 모든 조회에 `deleted_at IS NULL` 조건이 필요 — `@SQLRestriction`으로 자동 적용하지만, 네이티브 쿼리·QueryDSL 직접 작성 시 누락 위험.
- MySQL 부분 인덱스 미지원으로 인덱스 카디널리티에 삭제 데이터가 포함됨 — 90일 경과 행을 archive 스키마로 이전하는 배치가 결국 필요.
- UNIQUE 제약이 deleted 행도 포함하므로, 같은 이름으로 재생성하려면 deleted 행을 먼저 정리해야 할 수 있음 (운영 시점에 케이스별 대응).

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| 모든 테이블에 soft delete 적용 | 일관된 정책, 실수 방지 | 구조 편집이 빈번한 도메인의 인덱스·쿼리 부담. 의미상 "삭제 이력"이 가치 없는 곳까지 보존 비용 발생 |
| 모든 테이블에서 하드 삭제만 | 구조 단순, 인덱스 효율 | 사용자 복원 요구 대응 불가. 활동 추적 자료 손실 |
| 별도 `archive` 테이블에 이관 | 본 테이블 깨끗, 인덱스 효율 | 즉시 복원 비용 (조회→복사→삭제). 현재 규모에서 과도 |

## 다시 검토할 시점

- 데이터 규모가 활성 행 대비 deleted 행이 30% 이상으로 커지는 시점 — 90일 archive 배치 도입.
- PostgreSQL 또는 MySQL 부분 인덱스 지원 환경으로 이전하는 경우 — 부분 인덱스로 활성 데이터만 인덱싱.
- 새 도메인 추가 시 — 본 ADR의 자산성 vs 구조 편집 분류로 판단.
