# Rule: update-docs/dict 갱신 프로토콜

> 도메인 용어·고유 어휘 사전(Ubiquitous Language)을 기록한다.
> Story/Epic 종료 시 새로 등장하거나 의미가 확정된 용어만 일괄 반영한다.
> 본 규칙은 [`update-docs.md`](../update-docs.md) §1 패키지 일람의 dict 패키지 상세이다.

---

## 1. 위치·파일 구성

- **위치**: `update-docs/dict/`
- **현재 파일**: `Ubiquitous-Language.md` 1개 (현재 빈 상태 — 첫 항목부터 본 규칙에 따라 채운다).
- 파일 분리는 200개 항목 또는 BC별 페이지 분리 필요성이 보이기 전까지는 **단일 파일 유지**.

---

## 2. 갱신 트리거

작업 중 다음이 감지되면 **Story 종료 시 일괄 반영** 후보로 둔다.

- private-docs 또는 코드에서 **새 도메인 용어**가 처음 등장 (예: `OnFieldBudget`, `AxisTopic`).
- 기존 용어의 **의미가 명확히 정의되거나 변경**됨 (예: "ARCHIVE의 의미 = ...").
- 영문 ↔ 한글 매핑이 새로 확정됨 (UI/도메인 이름 매핑 충돌이 있을 때).
- 사용자가 "이 용어 사전에 박아줘"라고 명시한 경우 — 그땐 즉시.

---

## 3. 항목 포맷

```markdown
### {한글명} / {영문명}

- **소속 BC**: {Card / LearningFacade / ...}
- **정의**: {1-2문장. 도메인 안에서의 의미}
- **예/반례** (선택): ...
- **근거**: {private-docs/domain/{bc}.md §..., 또는 ADR{NNN}}
```

- 한글·영문 둘 다 기재. 한쪽만 알 수 있을 때는 사용자에게 묻고 매핑 확정 후 등록.
- **정의는 코드·구현 디테일이 아닌 도메인 의미**로 쓴다 — 필드 이름·테이블 이름 같은 표면형은 기재하지 않는다.
- 항목은 BC 그룹 → 가나다(한글) 또는 알파벳(영문) 순으로 정렬 (단일 파일 운용 시).

---

## 4. 갱신 절차

1. Story 작업 중 새 용어를 메모 (대화 컨텍스트로 충분).
2. Story 종료 push **직전**, 후보 용어 목록을 1줄씩 사용자에게 보고.
3. 사용자 승인 → `Ubiquitous-Language.md`에 항목 추가/수정.
4. 별도 커밋: `docs(dict): {용어 N개} 추가 [Story-{NNN}-{N}]`.

---

## 5. 수정 규칙

- 한 번 정의된 용어의 **정의 자체가 바뀌면** 기존 항목을 새 정의로 갱신하고, 변경 사유를 PR 본문에 1줄 적는다 (큰 변경이면 ADR 동반).
- 용어 폐기는 항목을 삭제하지 말고 `(deprecated, → 새 용어)`로 표시한다.
- 오타·표현 다듬기는 자유.

---

## 6. 커밋 규칙

- scope: `dict`
- 메시지 예: `docs(dict): OnFieldBudget, AxisTopic 정의 추가 [Story-{NNN}-{N}]`
- 한 Story에서 추가/수정된 모든 용어를 **한 커밋에 묶는다** — 용어별로 쪼개지 않는다.

---

## 7. 금지

- 코드에 등장만 했을 뿐 도메인적 의미가 부여되지 않은 식별자(헬퍼 클래스명, util 메서드명 등)는 등록 금지.
- 사용자에게 보고하지 않은 자동 일괄 등록 금지.
- 한쪽 언어만 정해진 채로 등록 금지 — 영문/한글 매핑 미상이면 사용자에게 묻는다.
- `private-docs/domain/용어사전.md` 와 본 파일은 별도 영역. 한쪽을 다른 쪽에 복사하지 않는다.
