# 도메인 모델 문서

---

<details>
<summary><strong>Deck</strong> — 학습 카드를 주제 단위로 묶어 관리하는 컨테이너</summary>

### 설명

Deck은 단순 폴더가 아니다.
학습자가 관련 카드들을 의미 있는 주제로 분류하고, 계층 구조를 통해 세분화할 수 있도록 돕는 학습 관리 단위다.

Deck은 카드 목록, 하위 덱, 소유 사용자, 공개 여부를 반드시 구분해서 가진다.

---

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 덱 식별자 |
| `name` | 덱 이름 |
| `lastAccessed` | 마지막 접근 시각 |
| `onLibrary` | 공개 라이브러리 등록 여부 (기본값 `false`) |
| `publishedAt` | 라이브러리 공개 시각 (정렬용) |
| `parentDeck` | 부모 덱 (최상위 덱이면 `null`) |
| `depth` | 계층 깊이 (최상위 = `0`) |
| `subDecks` | 하위 덱 목록 |
| `cards` | 소속 카드 목록 |
| `user` | 소유 사용자 |

---

### 행위

| 행위 | 설명 |
|------|------|
| `of(String name, Deck parentDeck, UserEntity user)` | 덱 생성 |
| `updateName(String name)` | 덱 이름 변경 |
| `updateLastAccessed()` | 마지막 접근 시각 갱신 |
| `updateParent(Deck newParent)` | 부모 덱 변경 및 depth 재계산 |
| `changeParent(Deck newParent)` | 부모 덱 변경 (`updateParent` 위임) |

---

### 규칙

- 덱 이름은 동일 사용자 내에서 중복될 수 없다.
- 덱 이름은 비어 있으면 안 된다.
- 최상위 덱의 `parentDeck`은 `null`이며, `depth`는 `0`이다.
- `depth`는 외부에서 주입하지 않고, `parentDeck.depth + 1`로 내부에서 계산한다.
- 부모 덱이 변경되면 `depth`도 반드시 재계산된다.
- 덱은 반드시 소유 사용자를 가진다.
- 카드 제거 시 고아 카드는 자동으로 삭제된다. (`orphanRemoval = true`)
- `onLibrary`는 덱 생성 시 기본적으로 비공개(`false`)이다.
- `publishedAt`은 공개 전에는 `null`이다.

---

### 검증

- `name`이 `null` 또는 blank이면 생성하거나 수정할 수 없다.
- `user`가 `null`이면 생성할 수 없다.
- `parentDeck == null`이면 `depth`는 반드시 `0`이어야 한다.
- `parentDeck != null`이면 `depth`는 반드시 `parentDeck.depth + 1`이어야 한다.
- 동일 `user_id` + 동일 `name` 조합은 DB 유니크 제약으로 보장한다.
- `lastAccessed`는 생성 시 현재 시각으로 초기화된다.

</details>

<summary><strong>Card BC</strong> — 회상 가능한 학습 단위의 생성·수정·상태 전환·관계 탐색</summary>

## Card (Aggregate Root)

회상 가능한 학습 단위를 모델링한다.

### 설명

Card는 단순 메모 저장소가 아니다.
학습자가 맥락을 보고, 단서를 통해 떠올리고, 마지막에 자기 언어로 핵심을 압축하도록 돕는 학습 단위다.

Card는 MainNote, KeywordCue, Summary, Tag, CardStatus를 반드시 구분해서 가진다.

---

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 카드 식별자 |
| `mainNote` | 학습 맥락 영역 |
| `summary` | 핵심 압축 영역 |
| `keywordCues` | 회상 단서 목록 (최소 1개) |
| `tags` | 연결 태그 목록 (최대 3개) |
| `status` | 현재 운영 위치 (`ON_FIELD` / `ARCHIVE`). 생성 시 `ON_FIELD`로 초기화 |
| `enteredFieldAt` | 현재 ON_FIELD 구간 진입 시각. 생성 시 현재 시각으로 초기화. ARCHIVE → ON_FIELD 복귀 시 재기록 |
| `viewCount` | 현재 ON_FIELD 구간에서의 학습 화면 노출 횟수. 생성 시 0으로 초기화. ON_FIELD 복귀 시 0으로 초기화 |

---

### 행위

| 행위 | 설명 |
|------|------|
| `create(MainNote, Summary, List<String> keywords, List<String> tagValues)` | 카드 생성. `enteredFieldAt = now()`, `viewCount = 0`으로 초기화 |
| `changeMainNote(String textContent, String imageUrl)` | MainNote 수정 |
| `changeSummary(String value)` | Summary 수정 |
| `replaceKeywords(List<String> values)` | Keywords 전체 교체 |
| `addKeyword(String value)` | 단서 추가 |
| `removeKeyword(Long keywordCueId)` | 단서 제거 |
| `addTag(Tag tag)` | 태그 연결 |
| `removeTag(Long tagId)` | 태그 제거 |
| `replaceTags(List<Tag> tags)` | 태그 전체 교체 |
| `archive()` | ARCHIVE 상태로 전환. 이미 `ARCHIVE`이면 무시 (멱등성 보장). `enteredFieldAt` / `viewCount` 보존 |
| `returnToField()` | ON_FIELD 상태로 복귀. 이미 `ON_FIELD`이면 무시 (멱등성 보장). `enteredFieldAt` 재기록, `viewCount` 0으로 초기화 |
| `incrementViewCount()` | ON_FIELD 상태일 때만 `viewCount + 1`. ARCHIVE 상태에서 호출 시 무시 |
| `isOnField()` | 현재 `ON_FIELD` 상태 여부 반환 |
| `isArchived()` | 현재 `ARCHIVE` 상태 여부 반환 |
| `isMaxViewReached(int maxView)` | `viewCount >= maxView` 여부 반환 |
| `isDurationExceeded(Duration maxDuration)` | `now() - enteredFieldAt >= maxDuration` 여부 반환 |

---

### 규칙

**학습 구조 규칙**
- 카드는 Main / Keywords / Summary를 분리해서 가진다.
- Main은 텍스트 또는 이미지 중 최소 하나를 포함해야 한다.
- Keywords는 최소 1개 이상이어야 한다.
- Summary는 비어 있으면 안 된다.
- Summary는 1~3문장 범위를 따라야 한다.
- 카드의 세 영역은 하나의 통합 텍스트로 저장하지 않는다.

**태그 규칙**
- 태그는 카드당 최대 3개까지만 연결할 수 있다.
- 태그는 카드 생성 시 연결하지 않아도 된다. (선택)
- 태그 제한(3개)은 "연결의 밀도를 높이기 위한 의도적 설계"다. 단순 기술적 제약이 아니다.

**상태 규칙**
- Card 생성 시 `status`는 반드시 `ON_FIELD`로 초기화된다.
- `archive()` / `returnToField()`는 내부 상태 전환만 수행한다.
- 이력 기록은 Card가 직접 책임지지 않는다. Application Service가 `CardStatusHistoryAppender`를 호출해 처리한다.
- 상태 전환 시 연결된 Tag, Deck 귀속 관계는 변경되지 않는다.

**ON_FIELD 체류 규칙**
- `enteredFieldAt`은 Card 생성 시, 그리고 ARCHIVE → ON_FIELD 복귀 시마다 현재 시각으로 재기록된다.
- `viewCount`는 ON_FIELD 구간 진입 시 0으로 초기화되며, `incrementViewCount()` 호출로만 증가한다.
- `incrementViewCount()`는 ON_FIELD 상태인 카드에만 유효하다. ARCHIVE 카드는 노출 대상이 아니므로 무시한다.
- `archive()` 시 `enteredFieldAt` / `viewCount`는 보존한다. 마지막 ON_FIELD 구간 기록은 통계 근거로 유지한다.
- 만료 조건 성립 여부에 따른 자동 Archive 전환은 Card가 직접 수행하지 않는다. `CardExpiryPolicy`가 판단하고 Application Service가 `archive()`를 호출한다.

---

### 검증

**생성 검증**
- `mainNote`가 `null`이면 생성할 수 없다.
- `summary`가 `null`이면 생성할 수 없다.
- `keywordCues`가 `null`이면 생성할 수 없다.
- `tags`가 `null`이면 생성할 수 없다. (빈 목록은 허용)
- `status`가 `null`이면 생성할 수 없다.
- 카드 생성 시 Keyword 목록은 최소 1개 이상이어야 한다.

**수정 검증**
- `replaceKeywords(...)` 호출 시 빈 목록으로 교체할 수 없다.
- `removeKeyword(...)` 수행 후 Keyword가 0개가 되면 안 된다.
- `addTag(...)` 호출 시 태그가 이미 3개이면 예외를 발생시킨다.
- `replaceTags(...)` 호출 시 4개 이상의 목록으로 교체할 수 없다.

**상태 전환 검증**
- `archive()` 호출 시 이미 `ARCHIVE` 상태이면 아무 작업도 수행하지 않는다.
- `returnToField()` 호출 시 이미 `ON_FIELD` 상태이면 아무 작업도 수행하지 않는다.
- `incrementViewCount()` 호출 시 ARCHIVE 상태이면 아무 작업도 수행하지 않는다.
- `isDurationExceeded()` 호출 시 `enteredFieldAt`이 `null`이면 `false`를 반환한다.
- `isMaxViewReached()` 호출 시 `maxView`가 0 이하이면 항상 `false`를 반환한다.
- Aggregate는 "회상 가능한 최소 구조"가 깨지지 않도록 생성/수정 시점마다 일관성을 검사해야 한다.

## MainNote (Value Object)

학습의 맥락과 근거를 담는 영역을 모델링한다.

### 설명

MainNote는 정답 자체를 그대로 적는 공간이 아니다.
학습 장면, 설명, 도식, 이미지, 배경 근거를 담는 역할을 한다.

### 속성

| 속성 | 설명 |
|------|------|
| `textContent` | 본문 텍스트 |
| `imageUrl` | 이미지 경로 |
| `contentType` | Main 구성 타입 (`TEXT_ONLY` / `IMAGE_ONLY` / `MIXED`) |

### 행위

| 행위 | 설명 |
|------|------|
| `of(String textContent, String imageUrl)` | MainNote 생성 |

### 규칙

- `textContent`와 `imageUrl`이 둘 다 비어 있으면 안 된다.
- 텍스트만 있을 수 있다.
- 이미지만 있을 수 있다.
- 텍스트와 이미지가 함께 있을 수 있다.
- 구성 상태에 따라 `contentType`이 내부에서 결정된다.

### 검증

- `textContent`, `imageUrl`은 저장 전에 trim 처리한다.
- `textContent`와 `imageUrl`이 모두 `null` 또는 blank이면 생성할 수 없다.
- 텍스트만 존재하면 `TEXT_ONLY`여야 한다.
- 이미지만 존재하면 `IMAGE_ONLY`여야 한다.
- 둘 다 존재하면 `MIXED`여야 한다.
- `contentType`은 외부에서 주입하지 않고 내부 규칙으로만 결정한다.

## KeywordCue (Entity)

회상을 유도하는 단서를 모델링한다.

### 설명

KeywordCue는 사용자가 정답을 바로 보는 대신,
짧은 단서를 보고 스스로 떠올리게 만드는 인출 장치다.

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 단서 식별자 |
| `value` | 단서 내용 |
| `card` | 소속 카드 |

### 행위

| 행위 | 설명 |
|------|------|
| `create(Card card, String value)` | 단서 생성 |

### 규칙

- Keyword는 비어 있으면 안 된다.
- 하나의 카드는 여러 개의 Keyword를 가질 수 있다.
- 카드 생성 시 최소 1개 이상 필요하다.
- Keyword는 카드에 종속된다.

### 검증

- `value`는 trim 처리 후 저장한다.
- `value`가 `null` 또는 blank이면 생성할 수 없다.
- `card`가 `null`이면 생성할 수 없다.
- 공백만 있는 문자열은 유효한 단서로 인정하지 않는다.
- 단건 Keyword는 자신의 값 유효성만 검사하고, 최소 개수 검사는 Card Aggregate가 담당한다.

---

## Summary (Value Object)

핵심을 자기 언어로 짧게 압축한 결과를 모델링한다.

### 설명

Summary는 원문 복붙이 아니라,
학습자가 이해한 핵심을 1~3문장으로 압축한 결과다.

### 속성

| 속성 | 설명 |
|------|------|
| `value` | 요약 내용 |

### 행위

| 행위 | 설명 |
|------|------|
| `of(String value)` | Summary 생성 |

### 규칙

- Summary는 비어 있으면 안 된다.
- Summary는 1~3문장 범위여야 한다.
- Summary는 너무 긴 설명문이 아니라 짧은 압축이어야 한다.
- 카드당 Summary는 하나만 가진다.

### 검증

- `value`는 trim 처리 후 저장한다.
- `value`가 `null` 또는 blank이면 생성할 수 없다.
- 문장 수는 1~3문장 범위여야 한다.
- 문장 수 판정 시 빈 문장은 제외한다.
- Summary는 단순 존재 여부만이 아니라 "짧은 압축"이라는 형태 규칙까지 함께 검사한다.

---

## CardStatus (Enum)

Card의 운영 위치를 모델링한다.

### 설명

CardStatus는 학습 성공·실패를 나타내는 평가 지표가 아니다.
"지금 이 카드가 전면 노출 구간에 있는가, 후방 대기 구간에 있는가"를 나타내는 운영 위치 개념이다.

익숙해져서 내려가든, 지금 당장 가치가 낮아서 내려가든 — 이유는 묻지 않는다. 위치만 기록한다.

### 값

| 값 | 설명 |
|----|------|
| `ON_FIELD` | 현재 전면 노출 구간. 지금 모르거나 익숙하지 않아 집중 반복이 필요한 카드를 올려두는 스테이징 공간 |
| `ARCHIVE` | 전면 루프에서 내려간 후방 대기 구간. 배경 지식으로 보관되며 언제든 재호출 가능 |

### 규칙

- Card는 생성 시 반드시 `ON_FIELD` 상태로 초기화된다.
- `ON_FIELD → ARCHIVE`, `ARCHIVE → ON_FIELD` 양방향 전환이 허용된다.
- 상태 전환은 `Card.archive()` / `Card.returnToField()` 행위를 통해서만 수행된다.
- 상태는 학습 달성 여부와 무관하다. 위치 판단은 학습자 의도에 따른다.

---

## CardStatusHistory (Entity)

Card의 운영 위치 전환 이력을 모델링한다.

### 설명

CardStatusHistory는 단순 로그가 아니다.
"이 카드가 언제 어디로 이동했는가"를 시계열로 보존해,
학습자의 카드 운영 패턴 분석과 재호출 판단의 근거를 제공한다.

Card Aggregate는 현재 위치(`status`)만 책임지고, 전환 이력의 보존은 CardStatusHistory가 담당한다.

수동 전환과 자동 전환(`CardExpiryPolicy`에 의한 만료 퇴장)을 이력 레벨에서 구분하지 않는다.
전환 사유 구분이 필요하면 v2에서 `reason` 필드(`MANUAL` / `MAX_VIEW` / `MAX_DURATION`)를 추가한다.

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 이력 식별자 |
| `card` | 이력이 속한 카드 |
| `fromStatus` | 전환 이전 위치 (`ON_FIELD` / `ARCHIVE`) |
| `toStatus` | 전환 이후 위치 (`ON_FIELD` / `ARCHIVE`) |
| `changedAt` | 전환이 발생한 시각 |

### 행위

| 행위 | 설명 |
|------|------|
| `of(Card card, CardStatus fromStatus, CardStatus toStatus)` | 이력 생성 |

### 규칙

- CardStatusHistory는 Card 상태 전환이 발생할 때마다 반드시 한 건 생성된다.
- `fromStatus`와 `toStatus`는 달라야 한다. 동일 상태 간 전환 이력은 생성하지 않는다.
- `changedAt`은 생성 시점에 자동 기록되며 변경할 수 없다.
- CardStatusHistory는 외부에서 직접 생성하지 않고 `CardStatusHistoryAppender`를 통해서만 생성된다.
- 이력은 삭제하지 않는다. Card가 삭제될 때만 함께 제거된다. (`orphanRemoval = true`)

### 검증

- `card`가 `null`이면 생성할 수 없다.
- `fromStatus`가 `null`이면 생성할 수 없다.
- `toStatus`가 `null`이면 생성할 수 없다.
- `fromStatus == toStatus`이면 생성할 수 없다.
- `changedAt`은 외부에서 주입하지 않는다.

---

## OnFieldBudget (Value Object)

ON_FIELD 구간의 체류 예산을 표현하는 값 객체다.

### 설명

OnFieldBudget은 "Card가 ON_FIELD에 얼마나 머물 수 있는가"를 하나의 결과로 표현한다.

현재 버전에서 예산은 시스템 수준에서 단일하게 관리된다.
Card별 예산 커스터마이징은 v2에서 별도 정책으로 분리한다.

### 속성

| 속성 | 설명 |
|------|------|
| `maxView` | 최대 허용 노출 횟수. 이 횟수에 도달하면 퇴장 조건이 성립한다 |
| `maxDuration` | 최대 허용 체류 기간 (`Duration`). 이 기간을 초과하면 퇴장 조건이 성립한다 |

### 행위

| 행위 | 설명 |
|------|------|
| `of(int maxView, Duration maxDuration)` | 예산 생성 |
| `isExpired(Card card)` | `isMaxViewReached` 또는 `isDurationExceeded` 중 하나라도 성립하면 `true` 반환 |

### 규칙

- `maxView`는 1 이상이어야 한다.
- `maxDuration`은 `null`이 아니어야 하며 양수여야 한다.
- 두 조건 중 하나라도 성립하면 퇴장 조건으로 간주한다. (OR 조건)
- `isExpired()`는 ARCHIVE 상태 카드에 대해서는 항상 `false`를 반환한다. 이미 퇴장한 카드는 만료 판단 대상이 아니다.

### 검증

- `maxView`가 1 미만이면 생성할 수 없다.
- `maxDuration`이 `null` 또는 0 이하이면 생성할 수 없다.

## CardTag (Entity)

Card와 Tag 사이의 연결을 명시적으로 관리하는 엔티티.

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 연결 식별자 |
| `card` | 연결된 카드 |
| `tag` | 연결된 태그 |
| `linkedAt` | 태그 연결 시각 |

### 규칙

- `card + tag` 조합은 유니크하다. (DB 복합 유니크 제약)
- `linkedAt`은 연결 시점에 자동 기록되며 변경할 수 없다.
- CardTag는 `Card.addTag()` / `Card.removeTag()`를 통해서만 생성·삭제된다. 외부에서 직접 생성하지 않는다.

## Tag (Entity)

카드와 카드 사이의 재회 통로 역할을 하는 연결 단위를 모델링한다.

### 설명

Tag는 단순 분류용 라벨이 아니다.
"이 카드와 연결된 다른 카드를 다시 만나게 하는 통로"로 작동한다.
리뷰 중 특정 주제를 마주쳤을 때, 관련 카드들을 태그를 통해 다시 찾아볼 수 있게 한다.
동일한 Tag를 가진 카드들은 서로 "관련 지식 후보"로 연결된다.

태그 개수를 카드당 최대 3개로 제한하는 이유는,
연결을 무분별하게 확산시키지 않고 핵심적인 연결만 남기기 위함이다.
태그가 많아질수록 연결은 희미해지고, 태그가 적을수록 연결은 진해진다.

### 속성

| 속성 | 설명 |
|------|------|
| `id` | 태그 식별자 |
| `value` | 태그 이름 |

### 행위

| 행위 | 설명 |
|------|------|
| `of(String value)` | 태그 생성 |

### 규칙

- 태그 이름은 비어 있으면 안 된다.
- 태그 이름은 시스템 전체에서 유니크하다.
- 태그는 카드에 종속되지 않는다. 여러 카드가 동일한 태그를 공유할 수 있다.
- 태그는 카드와 다대다(N:M) 관계다.
- 태그 삭제 시 연결된 카드에서도 해당 태그가 제거된다.
- 동일 태그를 가진 카드들은 "관련 지식 후보" 관계를 형성한다.
- 관련 카드 후보 조회 시 현재 카드 자신은 결과에서 제외된다.

### 검증

- `value`가 `null` 또는 blank이면 생성할 수 없다.
- `value`는 trim 처리 후 저장한다.
- 동일한 `value`를 가진 태그는 중복 생성할 수 없다. (DB 유니크 제약)

## RelatedCardCandidate (Value Object)

태그 기반으로 발견된 관련 카드 후보를 표현하는 값 객체다.

### 설명

RelatedCardCandidate는 단순히 카드를 담는 그릇이 아니다.
"어떤 태그를 통해 연결됐는가"를 함께 표현해,
후보가 왜 관련 있는지 근거를 명확히 드러낸다.

현재 카드와 태그를 1개 이상 공유하는 카드가 후보가 되며,
공유 태그 수가 많을수록 더 밀접한 연결로 간주한다.

### 속성

| 속성 | 설명 |
|------|------|
| `card` | 관련 카드 후보 |
| `sharedTags` | 현재 카드와 공유하는 태그 목록 (1개 이상) |
| `sharedTagCount` | 공유 태그 수 (정렬 기준). 내부에서 계산 |

### 행위

| 행위 | 설명 |
|------|------|
| `of(Card card, List<Tag> sharedTags)` | 관련 카드 후보 생성 |

### 규칙

- `sharedTags`는 반드시 1개 이상이어야 한다. (0개이면 후보가 아니다.)
- `card`는 현재 카드 자신이 아니어야 한다.
- `sharedTagCount`는 `sharedTags.size()`로 내부에서 계산한다. 외부 주입 불가.
- 후보 목록은 `sharedTagCount` 내림차순으로 정렬한다.
- `RelatedCardCandidate`는 외부에서 직접 생성하지 않고 `CardRelationFinder`를 통해서만 생성된다.

### 검증

- `card`가 `null`이면 생성할 수 없다.
- `sharedTags`가 `null` 또는 empty이면 생성할 수 없다.

## CardRelationFinder (Domain Service)

현재 카드를 기준으로 관련 카드 후보 목록을 생성하는 도메인 서비스다.

### 설명

CardRelationFinder는 "어떤 카드가 관련 후보인가"를 결정하는 도메인 규칙을 담는다.
태그가 공유되는 카드를 찾는 조회 자체는 Repository에 위임하고,
후보 구성과 정렬 규칙은 도메인 서비스가 책임진다.

### 행위

| 행위 | 설명 |
|------|------|
| `findCandidates(Card currentCard, List<Card> taggedCards)` | 현재 카드와 공유 태그를 가진 카드 목록을 `RelatedCardCandidate` 목록으로 변환 |

### 규칙

- 현재 카드 자신은 후보에서 제외한다.
- 공유 태그가 0개인 카드는 후보에서 제외한다.
- 후보 목록은 `sharedTagCount` 내림차순으로 정렬한다.
- 태그가 없는 카드(`tags`가 빈 목록)는 후보가 될 수 없다.
- ARCHIVE 카드도 관련 후보에 포함된다. (배경 지식 접근 허용)

## CardStatusHistoryAppender (Domain Service)

Card 상태 전환 시 이력을 생성하는 책임을 담는 도메인 서비스다.

### 설명

CardStatusHistoryAppender는 Card Aggregate와 CardStatusHistory 사이의
생성 책임을 분리하기 위해 존재한다.

Card Aggregate는 자신의 상태 전환 규칙만 책임지고,
이력 기록 여부와 생성 로직은 이 서비스가 담당한다.

수동 전환과 자동 전환(`CardExpiryPolicy`에 의한 만료 퇴장) 모두 이 서비스를 통해 이력이 기록된다.

### 행위

| 행위 | 설명 |
|------|------|
| `append(Card card, CardStatus fromStatus, CardStatus toStatus)` | 전환 이력을 생성하고 저장소에 기록한다 |

### 규칙

- `fromStatus == toStatus`이면 이력을 생성하지 않는다.
- 이력 생성 실패가 상태 전환 자체를 롤백하지 않는다. 이력은 부가 기록이며, 전환의 성공 여부에 영향을 주지 않는다. 트랜잭션 경계는 Application Service에서 결정한다.

### Application Service 처리 흐름
```java
CardStatus before = card.getStatus();
card.archive();                                     // 상태 전환만
CardStatus after  = card.getStatus();
historyAppender.append(card, before, after);        // 이력 기록 분리
```

## CardExpiryPolicy (Domain Service)

ON_FIELD 체류 예산 소진 여부를 판단하고 자동 Archive 전환을 처리하는 도메인 서비스다.

### 설명

CardExpiryPolicy는 "이 카드가 ON_FIELD에서 퇴장해야 하는가"를 결정하는 도메인 규칙을 담는다.
만료 판단 자체는 `OnFieldBudget.isExpired()`에 위임하고,
만료된 카드의 상태 전환 호출과 이력 기록 조율은 Application Service가 책임진다.

퇴장은 실패가 아닌 순환이다. 도메인 서비스는 퇴장 사유를 별도로 저장하지 않는다.
UX 문구("이 카드는 잠시 쉬러 갑니다")는 FE가 담당한다.

### 행위

| 행위 | 설명 |
|------|------|
| `expire(Card card, OnFieldBudget budget)` | `OnFieldBudget.isExpired(card)`가 `true`이고 카드가 ON_FIELD 상태이면 `card.archive()`를 호출한다. 실제 전환 발생 여부(`boolean changed`)를 반환한다 |

### 규칙

- 이미 ARCHIVE 상태인 카드에 대해서는 `expire()`를 수행하지 않는다. (멱등성 보장)
- `expire()` 내부에서 `card.archive()`를 호출한 경우, 이력 기록을 위해 Application Service가 `CardStatusHistoryAppender`를 이어서 호출해야 한다.

### Application Service 처리 흐름 (배치 기반 예시)
```java
List<Card> onFieldCards = cardRepository.findAllByStatus(ON_FIELD);

for (Card card : onFieldCards) {
    CardStatus before = card.getStatus();
    boolean changed   = expiryPolicy.expire(card, budget);
    CardStatus after  = card.getStatus();

    if (changed) {
        historyAppender.append(card, before, after);
        cardRepository.save(card);
    }
}
```

## CardReview (Entity) — 변경

리뷰 세션 안에서 카드 한 장의 공개 상태를 모델링한다.

### 설명

CardReview는 카드 자체가 아니다.
"이 카드가 지금 회상 단계인가, 비교 단계인가"를 추적하는 상태 단위다.

RECALLING 단계에서는 Main만 노출해 학습자가 먼저 스스로 떠올리게 한다.
COMPARING 단계에서는 Main과 Keywords·Summary를 함께 노출해
자신의 기억과 저장된 정답을 나란히 비교할 수 있게 한다.

### 속성

| 속성 | 설명 |
|------|------|
| `card` | 리뷰 대상 카드 (읽기 전용 참조) |
| `reviewStep` | 현재 공개 단계 (`RECALLING` / `COMPARING`) |
| `comparingStartedAt` | COMPARING 단계로 전환된 시각 (`null`이면 아직 전환 전) |

### 행위 — 변경

| 변경 전 | 변경 후 | 이유 |
|---------|---------|------|
| `reveal()` | `startComparing()` | "공개"가 아니라 "비교 시작"이라는 의도를 명확히 표현 |
| `isRevealed()` | `isComparing()` | 상태명 변경에 따른 일관성 유지 |
| `isMainVisible()` | 유지 | Main은 항상 노출. 변경 없음 |
| `isAnswerVisible()` | 유지 | COMPARING 상태일 때만 노출. 변경 없음 |
| 없음 | `visibleContent()` | **추가** — 현재 단계에서 노출할 영역을 하나의 결과로 반환 |

| 행위 | 설명 |
|------|------|
| `of(Card card)` | RECALLING 상태로 초기화 |
| `startComparing()` | COMPARING 상태로 전환 및 `comparingStartedAt` 기록 |
| `isComparing()` | COMPARING 상태 여부 반환 |
| `isMainVisible()` | Main 노출 여부 반환 (항상 `true`) |
| `isAnswerVisible()` | Keywords·Summary 노출 여부 반환 (`reviewStep == COMPARING`) |
| `visibleContent()` | 현재 단계 기준 노출 영역을 `CardVisibleContent`로 반환 |

### 규칙

- CardReview는 반드시 RECALLING 상태로 시작한다.
- Main은 RECALLING / COMPARING 모두에서 항상 노출된다.
- Keywords와 Summary는 COMPARING 상태일 때만 노출된다.
- COMPARING 단계에서는 Main과 Keywords·Summary가 반드시 함께 노출된다. 부분 공개는 없다.
- `RECALLING → COMPARING` 단방향 전환만 허용한다. 역방향 전환은 없다.
- `startComparing()`은 한 번만 유효하며, 이미 COMPARING이면 무시한다. (멱등성 보장)

### 검증

- `card`가 `null`이면 생성할 수 없다.
- `startComparing()` 호출 시 이미 COMPARING 상태이면 `comparingStartedAt`을 덮어쓰지 않는다.
- `visibleContent()`는 외부에서 노출 여부를 주입받지 않고 `reviewStep`만을 기준으로 결정한다.

### Deck Aggregate

> **결론: 애그리게이트 책임**

- 카드 목록과 하위 덱을 하나의 학습 컨테이너로 묶어 관리한다.
- 덱이 "유효한 학습 분류 단위" 구조를 유지하도록 전체 일관성을 보장한다.
- 계층 구조(self-reference)에서 `depth`가 항상 부모 기준으로 일관되게 유지되도록 강제한다.
- 부모 변경 시 `depth` 재계산을 내부에서 책임진다.
- 소유 사용자와의 연관 관계를 유지한다.
- 공개 라이브러리 등록 상태(`onLibrary`, `publishedAt`)를 관리한다.
- 마지막 접근 시각(`lastAccessed`)을 갱신해 활동 이력을 추적한다.

### Card Aggregate

> **결론: 회상 가능한 학습 단위 일관성 + 현재 위치 상태 + ON_FIELD 체류 추적 책임**

- 학습의 맥락(MainNote), 단서(KeywordCue), 압축(Summary)을 하나의 학습 단위로 묶어 일관성을 보장한다.
- "회상 가능한 최소 구조"(Main + Keywords + Summary)가 생성·수정 시 유지되도록 강제한다.
- 태그 연결 밀도 규칙(최대 3개)을 강제한다.
- 현재 운영 위치(`ON_FIELD` / `ARCHIVE`)를 보유하고 상태 전환 규칙(멱등성 포함)을 강제한다.
- `enteredFieldAt`을 통해 ON_FIELD 진입 시각을 추적한다.
- `viewCount`를 통해 현재 ON_FIELD 구간의 노출 횟수를 추적한다.
- ON_FIELD 복귀 시 두 필드를 초기화해 새 구간의 시작을 명확히 한다.
- 만료 조건 성립 여부를 `isMaxViewReached()` / `isDurationExceeded()`로 제공한다.
- 이력 기록과 자동 퇴장 판단은 책임지지 않는다.

### CardReview

> **결론: 카드 공개 상태 및 비교 흐름 책임**

- 카드 한 장의 공개 단계(RECALLING / COMPARING)를 추적한다.
- RECALLING 단계에서 Main만, COMPARING 단계에서 Main + Keywords·Summary를 함께 노출한다.
- 비교 시작 시각(`comparingStartedAt`)을 기록해 추후 통계 확장 근거를 보존한다.
- 단방향 상태 전환(RECALLING → COMPARING)을 강제해 역방향 흐름을 차단한다.
- `visibleContent()`를 통해 노출 영역 판단 책임을 내부에서 캡슐화한다.
