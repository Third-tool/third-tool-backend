# ADR006: 문서 체계를 docs/ 단일 진입으로 압축한다

- **상태**: Accepted
- **날짜**: 2026-05-14
- **관련**: [ADR005](ADR005.md), `.claude/rules/workflow.md`, `.claude/rules/review.md`

## 컨텍스트

문서 체계가 4개 영역(`private-docs/` / `update-docs/` / `.claude/rules/` / `docs/architecture/·database/`)에 분산되어 약 30,000줄·61개 파일에 이르렀다. 두 가지 관찰이 결정의 계기가 됐다.

1. **인용 부재** — 최근 30개 커밋 메시지에서 `근거:` 인용이 단 한 번도 사용되지 않았다. `private-docs/api/`·`table/`·`test/`의 reference 매칭 프로토콜이 실제 작업 흐름에서 활용되지 않는다는 신호.
2. **Plan mode 협업 전환** — Story 단위로 Plan mode 협업하면서, 사전에 reference 문서를 채워두는 모델보다 Plan mode가 매번 코드에서 추출하는 모델이 효율적이라는 판단.

또한 API 명세·테이블 스펙·테스트 매트릭스는 Controller / Flyway / 테스트 코드가 단일 진실 소스이므로 별도 문서 유지가 중복·동기화 부담을 만든다.

추가로 **Story 단위 Reviewer 세션**을 워크플로의 강제 단계로 도입할 필요가 생겼다 — 단일 Claude의 자가 점검으로는 도메인 캡슐화·BC 의존·NPE 가능성 등의 누락이 잦기 때문.

## 결정

문서 체계를 다음과 같이 단순화한다.

### tracked 영역 — GitHub 공개

```
docs/
├── DOMAIN.md         # 도메인 의도·용어·BC별 불변식 (단일 파일)
├── PACKAGE.md        # 패키지·BC·레이어 구조
└── adr/              # 결정 기록 (ADR{NNN}.md + index.md, tracked)
```

### untracked 영역 — 개발자만 보는 운영 자료

```
.claude/rules/
├── workflow.md       # Plan mode + Story + Reviewer 협업 흐름
├── pr-commit.md      # 브랜치·커밋·push·PR
├── conventions.md    # domain+api+db+test 통합 (4파일 → 1파일)
├── adr.md            # ADR 작성 트리거 (docs/adr/ tracked 정책)
└── review.md         # Story Reviewer 세션 (5관점 병렬 subagent)

workflow/             # 현재 진행 중인 Epic/Story (단발 운영)
```

**Reviewer 세션 도입**: workflow.md Step 4가 강제 단계로 신설된다. 5관점(Domain / Architecture / API·Exception / Test / Sceptical)을 `Agent` 도구의 병렬 subagent로 동시 발사하고, 메인 Claude가 종합해 사용자 의사결정으로 이어간다.

### 폐기

- `private-docs/` 전체 (16파일 / 약 8,000줄) — 도메인 의도는 `docs/DOMAIN.md`로 압축 이전, API·테이블·테스트는 코드가 진실 소스이므로 폐기.
- `update-docs/` 전체 (17파일 / 약 3,600줄) — ADR은 `docs/adr/` tracked로 이전, 나머지는 Plan mode가 매번 코드에서 추출.
- `docs/architecture/`, `docs/database/` (11파일 / 약 1,100줄) — `docs/PACKAGE.md`로 흡수.
- `.claude/rules/private-docs.md`, `update-docs.md`, `update-docs/*.md`, `{domain,api,db,test}-conventions.md` 13파일 → 5파일로 압축.

## 결과 (Consequences)

**긍정적**

- 문서 부담이 약 30,000줄 → 약 3,000줄로 1/10 압축. 유지 비용 감소.
- 진실 소스 명확화 — 도메인 의도는 `docs/DOMAIN.md`, 코드 사실은 코드 자체, 결정은 `docs/adr/`.
- Plan mode가 단일 진입(`docs/DOMAIN.md` + 코드)으로 필요한 컨텍스트를 빠르게 확보.
- ADR이 tracked로 GitHub에 공개되어 PR 리뷰·머지 흐름에 자연스럽게 합류.
- Reviewer 세션 강제로 단일 Claude 자가 점검의 누락을 5관점 병렬 검토로 보완.
- `.claude/` ignored 유지로 운영 룰은 개발자만 보는 영역에 둠. 내부 진행 상태는 GitHub 공개 차단.

**트레이드오프 / 부정적**

- BC별 도메인 상세 매트릭스(이전 `private-docs/domain/{BC}.md` 형식)가 사라짐 — 깊은 도메인 디테일은 `docs/DOMAIN.md`의 압축 본문에서 다루지 않음. 필요 시 코드 + 테스트로 보강.
- API 명세 문서가 사라져 외부 컨슈머 시점의 진입 자료는 Swagger UI에 의존 — 별도 API 문서 운영 부담이 Swagger·OpenAPI 측으로 이동.
- 테이블 스펙·테스트 매트릭스의 사후 점검은 Plan mode 의존 — 자동 일관성 점검 부재.
- Reviewer 세션 1회당 5배 토큰 비용 — Story당 1회 발생이지만 Epic당 3-5회.
- 직전에 `update-docs/adr/`가 한 차례 tracked로 GitHub에 노출됐던 history 흔적이 남음 (force push 없이 그대로 유지 — 사용자 결정).

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| 현재 4영역(private-docs / update-docs / docs / .claude/rules) 유지 | 변경 없음 | 30,000줄 유지 부담. 인용이 0건이라 reference 매칭 프로토콜이 실제 작동 안 함 |
| docs 전면 폐기 + 코드 주석/JavaDoc으로 도메인 의도 이전 | 가장 단순 | 코드만으론 BC 횡단 의도(예: "AxisTopic 명사구로 v4 전환 결정")를 알 수 없음. ADR도 함께 영속 가치 |
| private-docs 유지 + update-docs만 폐기 | 도메인·API·테이블 명세 별도 보존 | API·테이블은 코드가 진실 — 이중 관리 부담 유지. 인용 부재 문제 해결 안 됨 |
| ADR을 `update-docs/adr/` gitignored 유지 (이전 결정) | 내부 문서 GitHub 노출 차단 | 도메인 문서를 tracked로 결정한 의도와 일관성 손상. ADR이 PR 흐름에 합류되지 않으면 결정 가시성 부족 |

## 다시 검토할 시점

- **`docs/DOMAIN.md` 단일 파일이 1,500줄을 넘어가는 시점** — BC별 파일 분할(`docs/domain/{bc}.md`) 재도입 검토.
- **API 외부 컨슈머(다른 팀·외부 통합)가 생긴 시점** — Swagger UI 외에 별도 API 명세 문서 도입 검토.
- **Reviewer 세션 토큰 비용이 작업 흐름을 막을 정도가 된 시점** — 5관점 축소 또는 관점 통합 검토. 다만 §6 검증을 통해 누락 발견 가치를 정량화한 후에만.
- **Plan mode 사용량이 줄어 Story 단위 협업 모델이 변경되는 시점** — `private-docs/`처럼 사전 reference 모델 부분 복원 검토.
