# ADR002: Enum 컬럼은 VARCHAR + CHECK 제약으로 저장한다

- **상태**: Accepted
- **날짜**: 2026-05-07
- **관련**: [`private-docs/table/learningFacade.md`](../../private-docs/table/learningFacade.md) §3-3

## 컨텍스트

Enum 타입 도메인 값(`MaterialType`, `ProficiencyLevel`, `CoverageStatus`, `CardStatus`, `ArchiveReason`, `ReviewStep` 등)을 RDB에 어떤 형식으로 저장할지 결정해야 한다.

JPA의 기본 동작은 `@Enumerated(EnumType.ORDINAL)` — Enum 선언 순서대로 INT가 저장된다. 그러나 본 프로젝트는 도메인 모델 변경이 빈번하며(예: Epic 2의 `AxisAction → AxisTopic` 전환), Enum 값 추가·재배치가 발생할 수 있다.

운영 디버깅(MySQL Workbench, RDS 콘솔 직접 조회) 시 데이터의 의미가 즉시 읽혀야 한다.

## 결정

모든 Enum 컬럼은 다음 3요소를 함께 적용한다.

1. DB 컬럼: **`VARCHAR(N) NOT NULL`** (N은 enum 값의 최대 길이 + 여유. 보통 20).
2. DB 제약: **`CHECK (col IN ('VALUE_A', 'VALUE_B', ...))`** — 허용값 집합을 명시적으로 고정.
3. JPA 매핑: **`@Enumerated(EnumType.STRING)`**.

```java
@Enumerated(EnumType.STRING)
@Column(name = "coverage_status", length = 20, nullable = false)
private CoverageStatus coverageStatus;
```

```sql
CONSTRAINT chk_axis_topic_coverage
    CHECK (coverage_status IN ('NO_MATERIAL', 'PARTIAL', 'COVERED'))
```

## 결과

**긍정적**
- Enum 선언 순서가 바뀌어도 데이터가 오염되지 않는다.
- DB 콘솔에서 의미가 즉시 읽혀 운영·디버깅이 쉽다.
- CHECK 제약이 애플리케이션 검증을 빠뜨려도 잘못된 값 진입을 차단한다 (이중 방어).

**트레이드오프**
- INT(ordinal) 대비 저장 공간이 약간 더 필요. 인덱스도 더 큼 — 단, 카디널리티가 낮은 Enum이라 실측 영향은 미미.
- Enum 값 이름을 변경하려면 마이그레이션이 필요하다 (DDL 변경 + UPDATE).
- Enum 값 추가 시 모든 환경(dev/staging/prod)의 CHECK 제약을 갱신하는 Flyway 마이그레이션이 필요.

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| `@Enumerated(EnumType.ORDINAL)` (INT 저장) | 저장 공간 작음, 인덱스 효율 | Enum 선언 순서 변경 시 데이터 조용히 오염. 운영 콘솔에서 의미 안 보임 |
| 룩업 테이블 + FK | 정규화 형식, 메타데이터(라벨·설명) 부착 가능 | JOIN 비용. Enum이 도메인 코드와 1:1이라 독립 변경 사유 없음. 운영상 룩업 테이블 동기화 부담 |
| 자유 VARCHAR (CHECK 없음) | 마이그레이션 비용 작음 | 잘못된 값 진입 차단 안 됨. 도메인 검증 누락 시 데이터 오염 |

## 다시 검토할 시점

- Enum이 메타데이터(표시 라벨, 설명, 순서, 활성 여부)를 가져야 하는 경우 — 룩업 테이블로 전환 검토 (예: `RevisionReasonOption`이 이런 케이스).
- Enum 값이 수십 개 이상으로 커져서 CHECK 제약 갱신이 빈번해지는 시점.
