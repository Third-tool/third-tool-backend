# ADR001: Surrogate PK는 BIGINT AUTO_INCREMENT로 통일

- **상태**: Accepted
- **날짜**: 2026-05-07
- **관련**: [`private-docs/table/learningFacade.md`](../../private-docs/table/learningFacade.md) §3-2

## 컨텍스트

모든 BC(Card, Deck, LearningFacade, Review, User, UserSchedule)의 엔티티 PK 전략을 통일해야 한다.

운영 환경은 단일 RDS MySQL 8.0이며 샤딩 계획이 없다. REST API는 본인 리소스만 접근하는 구조(`/api/facade/me`, `/cards/{id}` 등)이고 소유권은 Application Service에서 검증된다 — 순차 ID가 URL에 노출되어도 열거 공격 표면이 좁다.

매핑 엔티티(`topic_material`, `card_tag`)에는 미래에 메모·적합도 점수 같은 속성이 추가될 가능성이 도메인 문서에 명시되어 있어, 복합 PK는 향후 FK 관계에서 부담이 된다.

## 결정

- 모든 테이블의 PK는 **`BIGINT NOT NULL AUTO_INCREMENT`**.
- 매핑 엔티티도 **surrogate `id` 컬럼**을 둔다 (`(topic_id, material_id)` 같은 복합 PK 사용 안 함).
- 외부 통합으로 public ID가 필요해지면 **`public_id CHAR(26)` 별도 컬럼**을 추가하고 내부 관계는 BIGINT 유지하는 하이브리드로 전환한다 — 지금은 도입하지 않는다.

JPA 매핑은 `@GeneratedValue(strategy = GenerationType.IDENTITY)` 통일.

## 결과

**긍정적**
- InnoDB 클러스터드 인덱스 친화: 순차 증가 BIGINT가 페이지 분할을 최소화.
- JPA 매핑·디버깅 단순.
- 매핑 엔티티 속성 확장이 자연스럽다.

**트레이드오프**
- BIGINT는 16바이트 ULID 대비 8바이트 절약. 외부 통합 시 별도 public ID 컬럼이 필요해질 수 있다.
- 순차 ID가 URL에 노출되므로 **소유권 검증을 Application Service에서 반드시 수행해야 한다** (이중 보호 없으면 IDOR 위험).

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| UUIDv4 (랜덤) | 분산 ID 생성, URL 열거 방지 | InnoDB PK 페이지 분할 빈발, 16바이트 PK + FK 전부 확대 비용. 분산 ID 요구 없음 |
| ULID / UUIDv7 (시간 정렬) | UUID 장점 + 시간 정렬로 클러스터드 인덱스 친화 | 동일 비용 발생, 외부 통합 요구가 없는 시점에 도입 가치 낮음 |
| 매핑 엔티티 복합 PK `(topic_id, material_id)` | PK 폭 절약 | 속성 확장 시 FK 관계가 불편. 도메인 문서가 미래 속성 확장을 명시 |

## 다시 검토할 시점

- 외부 시스템과의 통합으로 public ID 노출이 표준이 되는 시점 (CHAR(26) 하이브리드 도입 검토).
- 샤딩 도입 검토 시 (분산 ID 생성 전략 필요).
- 테이블 한 곳에 외부 통합용 public_id가 필요해지는 단발 케이스 발생 시 (그 테이블만 하이브리드 적용).
