# 도메인 모델 요약
---

## Card
회상 가능한 학습 단위를 모델링한다.

### 설명
Card는 단순 메모 저장소가 아니다.
학습자가 맥락을 보고, 단서를 통해 떠올리고, 마지막에 자기 언어로 핵심을 압축하도록 돕는 학습 단위다.

Card는 MainNote, KeywordCue, Summary를 반드시 구분해서 가진다.

### 속성
- `id` : 카드 식별자
- `mainNote` : 학습 맥락 영역
- `summary` : 핵심 압축 영역
- `keywordCues` : 회상 단서 목록

### 행위
- `create(MainNote, Summary, List<String>)` : 카드 생성
- `changeMainNote(String textContent, String imageUrl)` : Main 수정
- `changeSummary(String value)` : Summary 수정
- `replaceKeywords(List<String> values)` : Keywords 전체 교체
- `addKeyword(String value)` : 단서 추가
- `removeKeyword(Long keywordCueId)` : 단서 제거

### 규칙
- 카드는 Main / Keywords / Summary를 분리해서 가진다.
- Main은 텍스트 또는 이미지 중 최소 하나를 포함해야 한다.(여러개도 가능하다.)
- Keywords는 최소 1개 이상이어야 한다.
- Summary는 비어 있으면 안 된다.
- Summary는 1~3문장 범위를 따라야 한다.
- 카드의 세 영역은 하나의 통합 텍스트로 저장하지 않는다.

### 검증
- `mainNote`가 `null`이면 안 된다.
- `summary`가 `null`이면 안 된다.
- `keywordCues`가 `null`이면 안 된다.
- 카드 생성 시 Keyword 목록은 최소 1개 이상이어야 한다.
- `replaceKeywords(...)` 호출 시에도 빈 목록으로 교체할 수 없다.
- `removeKeyword(...)` 수행 후 Keyword가 0개가 되면 안 된다.
- Aggregate는 “회상 가능한 최소 구조”가 깨지지 않도록 생성/수정 시점마다 일관성을 검사해야 한다.

---

## MainNote
학습의 맥락과 근거를 담는 영역을 모델링한다.

### 설명
MainNote는 정답 자체를 그대로 적는 공간이 아니다.
학습 장면, 설명, 도식, 이미지, 배경 근거를 담는 역할을 한다.

### 속성
- `textContent` : 본문 텍스트
- `imageUrl` : 이미지 경로
- `contentType` : Main 구성 타입(TEXT_ONLY / IMAGE_ONLY / MIXED)

### 행위
- `of(String textContent, String imageUrl)` : MainNote 생성

### 규칙
- textContent와 imageUrl이 둘 다 비어 있으면 안 된다.
- 텍스트만 있을 수 있다.
- 이미지만 있을 수 있다.
- 텍스트와 이미지가 함께 있을 수 있다.
- 구성 상태에 따라 contentType이 결정된다.

### 검증
- `textContent`, `imageUrl`는 저장 전에 trim 처리한다.
- `textContent`와 `imageUrl`이 모두 `null` 또는 blank이면 생성할 수 없다.
- 텍스트만 존재하면 `TEXT_ONLY`여야 한다.
- 이미지만 존재하면 `IMAGE_ONLY`여야 한다.
- 둘 다 존재하면 `MIXED`여야 한다.
- `contentType`은 외부에서 주입하지 않고 내부 규칙으로만 결정한다.

---

## KeywordCue
회상을 유도하는 단서를 모델링한다.

### 설명
KeywordCue는 사용자가 정답을 바로 보는 대신,
짧은 단서를 보고 스스로 떠올리게 만드는 인출 장치다.

### 속성
- `id` : 단서 식별자
- `value` : 단서 내용
- `card` : 소속 카드

### 행위
- `create(Card, String value)` : 단서 생성

### 규칙
- Keyword는 비어 있으면 안 된다.
- 하나의 카드는 여러 개의 Keyword를 가질 수 있다.
- 카드 생성 시 최소 1개 이상 필요하다.
- Keyword는 카드에 종속된다.

### 검증
- `value`는 trim 처리 후 저장한다.
- `value`가 `null` 또는 blank이면 생성할 수 없다.
- `card`가 `null`이면 생성할 수 없다.
- 공백만 있는 문자열은 유효한 단서로 인정하지 않는다.
- 단건 Keyword는 자신의 값 유효성만 검사하고, 최소 개수 검사는 Card Aggregate가 담당한다.
---

## Summary
핵심을 자기 언어로 짧게 압축한 결과를 모델링한다.

### 설명
Summary는 원문 복붙이 아니라,
학습자가 이해한 핵심을 1~3문장으로 압축한 결과다.

### 속성
- `value` : 요약 내용

### 행위
- `of(String value)` : Summary 생성

### 규칙
- Summary는 비어 있으면 안 된다.
- Summary는 1~3문장 범위여야 한다.
- Summary는 너무 긴 설명문이 아니라 짧은 압축이어야 한다.
- 카드당 Summary는 하나만 가진다.

### 검증
- `value`는 trim 처리 후 저장한다.
- `value`가 `null` 또는 blank이면 생성할 수 없다.
- 문장 수는 1~3문장 범위여야 한다.
- 문장 수 판정 시 빈 문장은 제외한다.
- Summary는 단순 존재 여부만이 아니라 “짧은 압축”이라는 형태 규칙까지 함께 검사한다.

---

## MainContentType (enum)
MainNote의 구성 방식을 모델링한다.

### 값
- `TEXT_ONLY` : 텍스트만 존재
- `IMAGE_ONLY` : 이미지만 존재
- `MIXED` : 텍스트와 이미지가 함께 존재

### 규칙
- MainNote 생성 시 text/image 존재 여부에 따라 자동 결정된다.

### 검증
- enum 값은 외부 입력값으로 직접 결정하지 않는다.
- MainNote 내부 상태를 기준으로만 계산된다.
- 잘못된 조합 상태와 enum 값이 따로 놀지 않도록 항상 MainNote 생성 로직 안에서 함께 결정한다.

# 책임 분리

---

## Card Aggregate
### 책임
- Main / Keywords / Summary를 하나의 카드로 묶어 관리한다.
- 카드가 “회상 가능한 학습 단위” 구조를 유지하도록 전체 일관성을 보장한다.
- Main, Keywords, Summary가 서로 섞이지 않고 역할별로 분리되도록 관리한다.
- 카드 생성 시 최소한의 완전한 학습 구조를 갖추도록 강제한다.
- Keywords 목록의 교체, 추가, 제거 같은 변화 흐름을 통제한다.
- 카드가 최소 1개 이상의 Keyword를 유지하도록 보장한다.
- 회상 흐름의 중심 단위로서 내부 구성요소들을 조합하고 유지한다.

결론: **애그리게이트 책임**

---

## MainNote
### 책임
- 학습의 본문 맥락과 근거를 표현한다.
- 텍스트/이미지 기반 학습 재료를 관리한다.
- 텍스트와 이미지가 모두 비어 있는 잘못된 Main 생성을 막는다.
- Main의 구성 상태를 기준으로 content type을 결정한다.

결론: **학습 맥락 책임**

---

## KeywordCue
### 책임
- 회상을 유도하는 단일 단서를 표현한다.
- 공백 또는 무의미한 단서가 저장되지 않도록 막는다.
- 카드에 종속된 단서로서 자신의 값 일관성을 유지한다.

결론: **회상 단서 책임**

---

## Summary
### 책임
- 학습 내용을 짧게 압축한 결과를 표현한다.
- 비어 있는 요약을 막는다.
- 1~3문장 범위를 벗어나는 요약을 막는다.
- 긴 설명문이 아니라 핵심 압축 결과라는 성격을 유지한다.

결론: **핵심 압축 책임**

## Deck
학습 카드를 주제 단위로 묶어 관리하는 컨테이너를 모델링한다.

### 설명
Deck은 단순 폴더가 아니다.
학습자가 관련 카드들을 의미 있는 주제로 분류하고, 계층 구조를 통해 세분화할 수 있도록 돕는 학습 관리 단위다.

Deck은 카드 목록, 하위 덱, 소유 사용자, 공개 여부를 반드시 구분해서 가진다.

### 속성
- `id` : 덱 식별자
- `name` : 덱 이름
- `lastAccessed` : 마지막 접근 시각
- `onLibrary` : 공개 라이브러리 등록 여부 (기본값 `false`)
- `publishedAt` : 라이브러리 공개 시각 (정렬용)
- `parentDeck` : 부모 덱 (최상위 덱이면 `null`)
- `depth` : 계층 깊이 (최상위 = `0`)
- `subDecks` : 하위 덱 목록
- `cards` : 소속 카드 목록
- `user` : 소유 사용자

### 행위
- `of(String name, Deck parentDeck, UserEntity user)` : 덱 생성
- `updateName(String name)` : 덱 이름 변경
- `updateLastAccessed()` : 마지막 접근 시각 갱신
- `updateParent(Deck newParent)` : 부모 덱 변경 및 depth 재계산
- `changeParent(Deck newParent)` : 부모 덱 변경 (updateParent 위임)

### 규칙
- 덱 이름은 동일 사용자 내에서 중복될 수 없다.
- 덱 이름은 비어 있으면 안 된다.
- 최상위 덱의 `parentDeck`은 `null`이며, `depth`는 `0`이다.
- `depth`는 외부에서 주입하지 않고, 부모 덱의 `depth + 1`로 내부에서 계산한다.
- 부모 덱이 변경되면 `depth`도 반드시 재계산된다.
- 덱은 반드시 소유 사용자를 가진다.
- 카드 제거 시 고아 카드는 자동으로 삭제된다. (`orphanRemoval = true`)
- `onLibrary`는 덱 생성 시 기본적으로 비공개(`false`)이다.
- `publishedAt`은 라이브러리 공개 시각으로, 공개 전에는 `null`이다.

### 검증
- `name`이 `null` 또는 blank이면 생성하거나 수정할 수 없다.
- `user`가 `null`이면 생성할 수 없다.
- `depth`는 `parentDeck == null`이면 반드시 `0`이어야 한다.
- `depth`는 `parentDeck != null`이면 반드시 `parentDeck.depth + 1`이어야 한다.
- 동일 사용자(`user_id`) + 동일 이름(`name`) 조합은 DB 유니크 제약으로 보장한다.
- `lastAccessed`는 생성 시 현재 시각으로 초기화된다.

# 책임 분리

---

## Deck Aggregate
### 책임
- 카드 목록과 하위 덱을 하나의 학습 컨테이너로 묶어 관리한다.
- 덱이 "유효한 학습 분류 단위" 구조를 유지하도록 전체 일관성을 보장한다.
- 계층 구조(self-reference)에서 `depth`가 항상 부모 기준으로 일관되게 유지되도록 강제한다.
- 부모 변경 시 `depth` 재계산을 내부에서 책임진다.
- 소유 사용자와의 연관 관계를 유지한다.
- 공개 라이브러리 등록 상태(`onLibrary`, `publishedAt`)를 관리한다.
- 마지막 접근 시각(`lastAccessed`)을 갱신해 활동 이력을 추적한다.

결론: **애그리게이트 책임**

---

## 개선 제안

### 1. `of()` 시그니처의 미사용 파라미터
`scoringAlgorithmType`이 인자로 선언되어 있지만 내부에서 전혀 사용되지 않는다.
제거하거나 실제 필드로 반영해야 한다.

### 2. `changeParent` vs `updateParent` 중복
`changeParent`가 `updateParent`를 그대로 위임만 한다.
둘 중 하나로 통일하거나, 명확한 의도 차이가 있다면 주석으로 분리 이유를 명시해야 한다.

### 3. 하위 덱 `depth` 재귀 불일치 가능성
`updateParent`는 자신의 `depth`만 재계산하고, 하위 덱의 `depth`는 갱신하지 않는다.
부모 이동 시 서브덱 전체의 `depth`가 틀어질 수 있으므로,
서비스 레이어에서 하위 덱까지 재귀적으로 갱신하는 로직이 필요하다.
이 책임이 **서비스 레이어**에 있음을 명시해두면 혼선을 줄일 수 있다.

## UserEntity
회원을 식별하고 인증 방식을 관리하는 사용자 단위를 모델링한다.

### 설명
UserEntity는 단순 회원 정보 저장소가 아니다.
자체 로그인과 소셜 로그인(카카오/네이버)을 통합 관리하고, 계정 잠금·권한·소셜 연동 상태까지
일관되게 유지하도록 돕는 인증 관리 단위다.

UserEntity는 인증 방식(로컬/소셜), 계정 상태(잠금 여부), 권한(roleType), 소셜 연동 정보를
반드시 구분해서 가진다.

### 속성
- `id` : 사용자 식별자
- `username` : 로그인 식별자 (불변, 유니크)
- `password` : 비밀번호 (소셜 사용자는 더미값 `"SOCIAL_USER"` 고정)
- `isLock` : 계정 잠금 여부
- `isSocial` : 소셜 로그인 계정 여부
- `socialProviderType` : 소셜 제공자 종류 (카카오/네이버, 자체 로그인이면 `null`)
- `roleType` : 사용자 권한 (USER / ADMIN)
- `nickname` : 표시 이름
- `email` : 이메일 주소
- `createdDate` : 계정 생성 시각
- `updatedDate` : 마지막 수정 시각
- `kakaoMembers` : 카카오 소셜 연동 목록
- `naverMembers` : 네이버 소셜 연동 목록

### 행위
- `ofLocal(username, encodedPassword, nickname, email)` : 자체 로그인 회원 생성
- `ofSocial(username, socialProviderType, nickname, email)` : 소셜 로그인 회원 생성
- `of(...)` : 전체 파라미터 기반 범용 생성 (내부/테스트용)
- `updateUser(UserUpdateRequestDTO)` : 닉네임·이메일 수정

### 규칙
- `username`은 생성 이후 변경할 수 없다.
- `password`는 자체 로그인 사용자는 인코딩된 값, 소셜 사용자는 `"SOCIAL_USER"` 더미값으로 고정한다.
- 소셜 계정은 반드시 `isSocial = true`이고 `socialProviderType`이 존재해야 한다.
- 자체 로그인 계정은 반드시 `isSocial = false`이고 `socialProviderType`은 `null`이어야 한다.
- 신규 계정은 기본적으로 `isLock = false`(잠금 해제) 상태로 생성된다.
- 신규 계정은 기본적으로 `roleType = USER`로 생성된다.
- 소셜 연동(`kakaoMembers`, `naverMembers`)이 삭제되면 연관 레코드도 함께 제거된다. (`orphanRemoval = true`)
- 도메인 객체 생성은 반드시 정적 팩토리 메서드(`ofLocal`, `ofSocial`)를 통해서만 허용한다.

### 검증
- `username`이 `null` 또는 blank이면 생성할 수 없다.
- `password`가 `null`이면 생성할 수 없다.
- `isSocial = true`인데 `socialProviderType`이 `null`이면 안 된다.
- `isSocial = false`인데 `socialProviderType`이 존재하면 안 된다.
- `roleType`이 `null`이면 생성할 수 없다.
- `isLock`이 `null`이면 생성할 수 없다.

## SocialMember (추상)
소셜 제공자별 연동 정보를 모델링하는 베이스 엔티티다.

### 설명
SocialMember는 카카오·네이버처럼 소셜 제공자마다 달라지는 인증 정보를 표현한다.
하나의 UserEntity에 여러 소셜 연동이 연결될 수 있으며, 제공자별로 구체 타입이 분리된다.

### 속성
- `id` : 연동 식별자
- `user` : 소속 사용자
- `socialId` : 소셜 제공자가 발급한 고유 식별자
- `refreshToken` : 소셜 제공자 리프레시 토큰
- `providerType` : 소셜 제공자 종류 (구체 타입이 반환)

### 행위
- `getProviderType()` : 구체 타입이 구현하는 제공자 식별 (추상)
- `updateRefreshToken(String refreshToken)` : 리프레시 토큰 갱신

### 규칙
- `socialId`는 소셜 제공자 내에서 유니크해야 한다.
- `user`가 `null`이면 생성할 수 없다.
- `socialId`가 `null` 또는 blank이면 생성할 수 없다.
- 구체 타입(KakaoMember / NaverMember)은 반드시 `getProviderType()`을 구현해야 한다.
- JPA 상속 전략은 JOINED이며, `provider_type` 컬럼이 구분자 역할을 한다.

### 검증
- `socialId`는 DB 유니크 제약으로 중복을 보장한다.
- `refreshToken`은 갱신 가능하며, 초기값이 `null`일 수 있다.

## CustomOAuth2User
Spring Security OAuth2 인증 흐름에서 UserEntity를 감싸는 어댑터를 모델링한다.

### 설명
CustomOAuth2User는 도메인 객체가 아니다.
Spring Security의 `OAuth2User` 인터페이스를 구현해, 인증 컨텍스트에서 UserEntity를 안전하게
노출하는 어댑터 역할을 한다.
소셜 제공자의 원본 응답(`attributes`)과 도메인 사용자(`user`)를 함께 보관한다.

### 속성
- `user` : 도메인 사용자 엔티티
- `attributes` : 소셜 제공자 원본 응답 데이터 (카카오/네이버 API 응답)

### 행위
- `getAttributes()` : 소셜 제공자 원본 응답 반환
- `getAuthorities()` : `ROLE_{roleType}` 형태의 권한 목록 반환
- `getName()` : Security 컨텍스트 기준 식별자 반환 (`username`)
- `getUsername()` / `getEmail()` / `getNickname()` / `isSocial()` : UserEntity 편의 접근자

### 규칙
- 권한은 `UserEntity.roleType`을 기반으로 `ROLE_` 접두어를 붙여 결정한다.
- `attributes`는 소셜 제공자 응답 원본을 그대로 보관하며, 직접 가공하지 않는다.
- `CustomOAuth2User`는 인증 흐름에서만 사용하고, 비즈니스 로직에 직접 전달하지 않는다.

---

## SocialProviderType (enum)
지원하는 소셜 로그인 제공자 종류를 모델링한다.

### 값
- `KAKAO` : 카카오
- `NAVER` : 네이버

### 규칙
- 소셜 로그인 회원 생성 시 반드시 지정되어야 한다.
- 자체 로그인 계정에는 존재하지 않는다 (`null`).
- `UserEntity`와 `SocialMember` 양쪽 모두에서 사용한다.

---

## UserRoleType (enum)
사용자 권한 수준을 모델링한다.

### 값
- `USER` : 일반 사용자
- `ADMIN` : 관리자

### 규칙
- 신규 계정 생성 시 기본값은 `USER`다.
- `ADMIN`은 외부 가입 흐름으로 부여되지 않는다.
- Spring Security 권한 문자열은 `ROLE_{UserRoleType}` 형태로 변환된다.

## ReviewSession
카드 목록을 순서대로 리뷰하는 학습 세션 단위를 모델링한다.

### 설명
ReviewSession은 단순 카드 목록 조회가 아니다.
학습자가 "회상 → 확인" 흐름을 순서대로 지키며 카드를 리뷰할 수 있도록
흐름 자체를 강제하는 학습 진행 단위다.

ReviewSession은 리뷰 대상 카드 목록, 현재 진행 위치, 각 카드의 공개 상태를
반드시 구분해서 가진다.

### 속성
- `id` : 세션 식별자
- `user` : 세션 소유 사용자
- `deck` : 리뷰 대상 덱
- `cardReviews` : 리뷰 대상 카드 상태 목록 (순서 보장)
- `currentIndex` : 현재 리뷰 중인 카드 위치
- `startedAt` : 세션 시작 시각

### 행위
- `start(Deck deck, UserEntity user)` : 세션 생성 및 리뷰 순서 초기화
- `currentCardReview()` : 현재 리뷰 중인 CardReview 반환
- `revealCurrentCard()` : 현재 카드를 REVEALED 상태로 전환
- `nextCard()` : 다음 카드로 이동 (REVEALED 상태일 때만 허용)
- `isFinished()` : 모든 카드 리뷰 완료 여부 반환

### 규칙
- 세션은 반드시 하나 이상의 카드를 포함해야 한다.
- 첫 번째 카드는 세션 시작 시 자동으로 RECALLING 상태로 초기화된다.
- `nextCard()`는 현재 카드가 REVEALED 상태일 때만 호출할 수 있다.
- RECALLING 상태에서 다음 카드로 건너뛸 수 없다.
- 마지막 카드 이후 `nextCard()`를 호출하면 안 된다.
- 리뷰 흐름은 반드시 "회상 → 확인" 순서를 따른다.

### 검증
- `deck`이 `null`이면 생성할 수 없다.
- `user`가 `null`이면 생성할 수 없다.
- `deck` 내 카드가 0개이면 세션을 시작할 수 없다.
- `revealCurrentCard()` 호출 시 이미 REVEALED 상태이면 다시 전환하지 않는다.
- `nextCard()` 호출 시 현재 카드가 RECALLING 상태이면 예외를 발생시킨다.
- `nextCard()` 호출 시 `isFinished() == true`이면 예외를 발생시킨다.

---

## CardReview
리뷰 세션 안에서 카드 한 장의 공개 상태를 모델링한다.

### 설명
CardReview는 카드 자체가 아니다.
"이 카드가 지금 회상 단계인가, 확인 단계인가"를 추적하는 상태 단위다.
리뷰 흐름에서 Main / Keywords·Summary의 공개 여부를 결정하는 기준이 된다.

### 속성
- `card` : 리뷰 대상 카드
- `reviewStep` : 현재 공개 단계 (RECALLING / REVEALED)
- `revealedAt` : Keywords·Summary가 공개된 시각 (`null`이면 아직 공개 전)

### 행위
- `of(Card card)` : RECALLING 상태로 초기화
- `reveal()` : REVEALED 상태로 전환 및 `revealedAt` 기록
- `isRevealed()` : REVEALED 상태 여부 반환
- `isMainVisible()` : Main 노출 여부 반환 (항상 `true`)
- `isAnswerVisible()` : Keywords·Summary 노출 여부 반환 (`reviewStep == REVEALED`)

### 규칙
- CardReview는 반드시 RECALLING 상태로 시작한다.
- Main은 항상 노출된다. RECALLING 상태에서도 숨기지 않는다.
- Keywords와 Summary는 REVEALED 상태일 때만 노출된다.
- RECALLING → REVEALED 전환만 허용한다. 역방향 전환은 없다.
- `reveal()`은 한 번만 유효하며, 이미 REVEALED이면 무시한다.

### 검증
- `card`가 `null`이면 생성할 수 없다.
- `reveal()` 호출 시 이미 REVEALED 상태이면 `revealedAt`을 덮어쓰지 않는다.
- `isAnswerVisible()`은 `reviewStep`만을 기준으로 판단하며, 외부에서 공개 여부를 직접 주입할 수 없다.

---

## ReviewStep (enum)
카드 리뷰의 공개 단계를 모델링한다.

### 값
- `RECALLING` : Main만 노출된 상태. 학습자가 스스로 회상을 시도하는 단계
- `REVEALED` : Keywords·Summary까지 모두 공개된 상태. 정답을 확인하는 단계

### 규칙
- 모든 CardReview는 반드시 RECALLING으로 시작한다.
- RECALLING → REVEALED 단방향 전환만 허용한다.
- ReviewStep은 외부에서 직접 주입하지 않고, `reveal()` 행위를 통해서만 전환된다.

# 개선 제안

### 1. ReviewSession과 Deck의 결합도
현재 설계에서 ReviewSession은 Deck을 직접 참조한다.
라이브러리에서 Fork한 덱이나 단일 카드 리뷰 같은 확장 시나리오가 생기면
`deck` 대신 `List<Card>`를 직접 받는 형태로 변경할 수 있다.
현 시점에서는 Deck 기반으로 단순하게 유지하고, 확장 필요 시 분리를 검토한다.

### 2. 카드 순서 전략의 분리
현재 `start()` 내부에서 카드 순서를 결정하는데,
순서 전략(등록 순 / 랜덤 / 오답 우선 등)이 추가될 경우 `ReviewSession` 내부가 복잡해진다.
순서 결정 책임을 `CardOrderStrategy` 같은 별도 전략 객체로 분리하면
`ReviewSession`이 흐름 제어에만 집중할 수 있다.
현 MVP에서는 등록 순 고정으로 단순하게 유지한다.

### 3. `revealedAt` 활용 범위 명시
`revealedAt`은 현재 단순 기록용이지만,
추후 "회상에 걸린 시간" 기반 난이도 조정이나 통계 기능을 붙이려면
`startedAt`(카드 노출 시각)도 함께 기록해야 한다.
지금은 `revealedAt`만 기록하고, 통계 기능 도입 시 `startedAt` 추가를 검토한다.
