# Rule: update-docs/adr 갱신 프로토콜

> Architecture Decision Record(ADR)의 작성 권한·트리거·형식을 정의한다.
> ADR은 [`update-docs.md`](../update-docs.md)가 정의한 갱신 영역(`update-docs/`)의 패키지 중 하나이며, 본 문서는 그 중 ADR 패키지의 트리거·절차·템플릿을 구체화한다.
> 갱신 시점은 **즉시 트리거** — Story/Epic 종료를 기다리지 않고 결정이 일어난 순간 작성한다 (`update-docs.md` §3 트리거 분류 참조).

---

## 1. 위치·명명·번호

- **위치**: `update-docs/adr/`
- **파일명**: `ADR{NNN}.md` (3자리 zero-padded, 1부터 순차)
- **인덱스**: `update-docs/adr/index.md` — 새 ADR 추가 시 자동으로 한 줄 append
- **한 ADR = 한 결정**. 하나의 파일에 여러 결정 묶지 않는다.
- **번호는 절대 재할당하지 않는다.** Superseded되어도 파일은 보존하고 상태만 변경한다.

---

## 2. 자동 작성 트리거 (작업 중 감지 시)

작업 중 다음 조건이 **둘 이상** 동시에 충족되면 ADR을 자동 작성한다.

- 결정이 **단일 메서드/파일을 넘어** 여러 BC 또는 레이어에 영향을 준다.
- **현실적 대안이 2개 이상** 존재했고, 명확한 거부 사유가 있다.
- "왜 X로 했는가"를 6개월 뒤에도 묻게 될 가능성이 있다.
- 미래에 같은 영역을 만지는 사람이 **이 결정을 인지하지 않으면 안 되는** 구속력이 있다.
- 결정의 전제가 깨질 만한 미래 시점이 식별 가능하다 (예: "샤딩 도입 시", "유저 100만 명 초과 시").

**감지 방식**: Story 실행 중, 코드 검토 중, 또는 사용자와의 대화 중에 위 패턴이 보이면 작성 권한을 자동 행사한다. 작성 전 사용자 동의를 받지 않는다 — 작성 후 한 줄로 보고한다.

---

## 3. 작성 금지 (이건 ADR이 아니다)

- 한 함수 / 한 파일 안에서 끝나는 구현 선택
- 코드·테스트만 봐도 명백한 결정 (네이밍, 시그니처 같은)
- Story 범위 안의 작업 결정 — 이건 Story 문서로
- 이미 있는 ADR이 다루는 영역의 세부 — 그 ADR을 보강(주석 갱신)하거나 Supersede
- 임시 우회·땜빵 — 이건 코드 주석으로
- BC 내부의 단일 도메인 모델링 결정 (도메인 문서가 1차 진실)

ADR을 쓸지 말지 모호하면 **쓰지 않는다.** ADR은 희소해야 가치가 있다.

---

## 4. ADR 표준 템플릿

```markdown
# ADR{NNN}: {결정의 한 줄 요약}

- **상태**: {Proposed | Accepted | Superseded by ADR{X} | Deprecated}
- **날짜**: YYYY-MM-DD
- **관련**: {연관 ADR 번호 또는 외부 문서 경로 — 없으면 생략}

## 컨텍스트

{이 결정이 필요해진 상황. 무엇이 문제였는가. 어떤 제약·요구사항이 있었는가. 왜 지금 결정해야 하는가.}

## 결정

{무엇을 정했는가. 명령형으로 짧게. 한 문단 또는 짧은 불릿.}

## 결과 (Consequences)

**긍정적**
- ...

**트레이드오프 / 부정적**
- ...

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| ... | ... | ... |

## 다시 검토할 시점

{이 결정의 전제가 깨지는 조건. 예: "샤딩 도입 시", "유저 수 100만 명 초과 시", "외부 통합 요구 발생 시". 없으면 "재검토 시점 없음 (구조적 결정)"이라고 적는다.}
```

---

## 5. 자동 작성 워크플로

§2 트리거가 감지되면:

1. **작업을 멈추지 않고** 새 `ADR{NNN}.md` 파일을 작성한다 (다음 비어있는 번호).
2. `index.md`에 한 줄 append한다 (§7 형식).
3. 코드 변경이 있는 작업 안에서 발생했다면 **ADR을 별도 커밋으로 분리**한다.
   - 커밋 메시지: `docs(adr): ADR{NNN} {짧은 요약} [Story-{n}]`
   - scope는 항상 `adr`. BC 이름이 아니다.
4. PR 본문 "참고 사항"에 새 ADR 번호와 1줄 요약을 적는다.
5. 사용자에게 한 줄 보고 — "ADR{NNN} 추가: {요약}" 정도. 사후 통지이지 사전 승인 요청이 아니다.

---

## 6. ADR 수정 규칙

- **Accepted된 ADR의 결정·컨텍스트는 수정하지 않는다.** 결정 내용을 바꾸려면 새 ADR을 만들고 기존 ADR의 상태를 `Superseded by ADR{X}`로 갱신한다.
- 상태 갱신 시 `index.md`의 해당 행도 동시 갱신한다.
- 오타·표현 다듬기·"관련" 메타데이터(연관 ADR 링크) 추가는 허용 — 결정의 의미가 바뀌지 않는 범위에서만.
- ADR 삭제는 금지. 폐기되었으면 상태를 `Deprecated`로 둔다.

---

## 7. index.md 형식

```markdown
# ADR Index

| # | 제목 | 상태 | 날짜 |
| --- | --- | --- | --- |
| [ADR001](./ADR001.md) | ... | Accepted | YYYY-MM-DD |
| [ADR002](./ADR002.md) | ... | Superseded by ADR007 | YYYY-MM-DD |
```

- 신규 ADR 추가 시 표 끝에 한 줄 append.
- 상태 변경(특히 `Superseded by ADR{X}`) 시 해당 행을 갱신.
- 표 외에 다른 내러티브를 두지 않는다 — 인덱스는 순수 인덱스다.

---

## 8. 작성 시 주의

- **결정 직후의 톤으로 쓴다.** "예전엔 X였는데 지금은 Y"가 아니라 "Y로 정한다"로.
- **컨텍스트는 짧게.** 결정 당시의 제약·요구사항만. 도메인 설명을 늘어놓지 않는다.
- **거부 사유를 반드시 적는다.** "고려했지만 거부함"이라는 사실 자체가 ADR의 핵심 가치다.
- **중립적으로 쓴다.** 거부된 대안도 무시·비하하지 않는다 — 같은 상황을 다시 검토할 사람을 위한 글이다.
- **코드 인용은 최소화.** 코드는 변하지만 결정의 이유는 영속이어야 한다.
