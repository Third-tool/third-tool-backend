# ADR004: 학습 주제는 명사구 표현으로 정의한다 — 단일 동사 강제 폐지

- **상태**: Accepted
- **날짜**: 2026-05-07
- **관련**: [`private-docs/domain/learning-facade.md`](../../private-docs/domain/learning-facade.md), [`private-docs/table/learningFacade.md`](../../private-docs/table/learningFacade.md) §3-6

## 컨텍스트

구 `AxisAction`은 학습 단위를 "단일 동사"로 표현하도록 강제했다 (`Description.validateSingleVerb()`, `VerbFormValidator`). 의도는 "행위로서의 학습"을 명확히 한다는 것이었지만 운영 결과:

- 유저가 자신의 학습 맵을 자유롭게 작성하지 못함. "REST vs GraphQL 선택 기준" 같은 주제는 단일 동사로 표현 불가.
- 동사 변경 시 `coverage_status`가 `NO_MATERIAL`로 강제 초기화되어 기존 자료 연결이 의미 없이 끊김.
- `revisionCount`, `ActionRevision`, `ActionChangeRecord` 등 동사 변경 이력 모델이 도메인을 복잡하게 만듦.

이 제약이 도메인 모델 자체에 깊이 박혀 있어 단순 검증 완화로는 풀리지 않고, 엔티티 자체를 재정의해야 했다.

## 결정

학습 단위를 **`AxisTopic`(명사구 기반 주제)**로 재정의한다.

- `AxisTopic.name` (필수, `VARCHAR(100)`): 짧은 명사구. 자유 형식. 공백 포함 허용.
- `AxisTopic.description` (선택, `VARCHAR(500)`, nullable): 부연 설명. 빈 문자열은 `null`로 정규화.
- 단일 동사 강제 검증 **폐지**.
- 동사 변경 → `NO_MATERIAL` 강제 초기화 **폐지**. 주제 `name`/`description` 수정은 커버리지 재계산 트리거가 아니다 — 주제 내용이 바뀌어도 기존 자료 연결의 유효성은 유저가 판단한다.
- 커버리지 재계산 트리거는 두 가지로 단순화: `TopicMaterial` 연결/해제 + `LearningMaterial.proficiencyLevel` 변경.

**구 모델 처리**:
- `AxisAction` 엔티티, `Description` VO, `VerbFormValidator` 제거.
- `action_material` 테이블 → `topic_material`로 RENAME.
- `action_revision` 테이블 → `axis_action_description_history`로 RENAME 후 아카이브 (FK 제거, 신규 코드 미참조).
- `revision_reason_option` 테이블은 삭제하지 않고 보존 (Epic 3 재사용 대기).
- `revisionCount`, `isRefinementSuggested()`, `ActionChangeRecord`는 Epic 3 `TopicRevision` 도입 시점으로 이관.

마이그레이션은 Epic 2 Story 2-0에서 완료 (Forward + Rollback 스크립트 포함).

## 결과

**긍정적**
- 유저 학습 맵 표현 자유도 회복. 명사구·문장형 주제 모두 허용.
- 도메인 모델 단순화: 단일 동사 검증 로직, 동사 변경 이력, 커버리지 강제 초기화 모두 제거.
- 커버리지 재계산 트리거가 명확해짐 (자료 연결·해제 + 숙련도 변경 두 가지로 응집).
- `description` 부연 설명 필드 추가로 짧은 제목 + 긴 설명 분리.

**트레이드오프**
- 동일 축 내 주제 이름 중복이 도메인 레벨에서 차단되지 않음 — UX 안내 책임이 FE로 이동 (도메인 명세가 의도적으로 허용).
- 주제 수정 이력이 Epic 2 시점에는 보존되지 않음 — Epic 3 `TopicRevision` 도입 전까지 회고 자료 공백.
- 기존 `axis_action.description`(30자 제약)이 `axis_topic.name`(100자)으로 확장되어 롤백 시 내용 잘림 위험 (롤백 스크립트에 `SUBSTRING` 처리 + 수동 확인 절차 명시).

## 대안 비교

| 대안 | 장점 | 거부 사유 |
| --- | --- | --- |
| 동사 강제 유지 + `description` 추가 | 모델 변경 최소 | 핵심 문제(자유 표현 차단)가 풀리지 않음. 동사 변경 시 강제 초기화 부작용도 그대로 |
| `AxisAction` 유지하되 `description.validateSingleVerb()`만 폐지 | 점진적 변경 | 엔티티 이름·테이블 이름이 더 이상 도메인을 표현하지 않게 됨. `revisionCount`·`ActionRevision`도 계속 어울리지 않음 |
| 주제 이름 중복을 도메인에서 차단 | 데이터 정합성 강화 | 명사구는 동의어·유사 표현이 자연스럽다 (예: "인덱스 설계" vs "인덱스 설계 원칙"). 차단은 표현 자유를 다시 제약 |

## 다시 검토할 시점

- Epic 3에서 `TopicRevision` 도입 시 — `revision_reason_option`을 재연결하고, `name`/`description` 수정 이력 추적을 추가하면서 본 ADR의 "커버리지 재계산 트리거" 단순화 결정을 재확인한다.
- 동일 축 내 중복 주제 이름이 운영 데이터에서 문제로 부각되는 시점 — UX 안내가 부족하다고 판단되면 도메인 검증 도입 재검토.
- `axis_action_description_history` 아카이브의 활용 빈도가 6개월 이상 0인 시점 — DROP 또는 cold storage 이관.
